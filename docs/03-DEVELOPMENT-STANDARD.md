# Development Standard

## 基本原則

### 禁止超大型單一 Worker

不得把身份、權限、商業流程、排程、Webhook 與資料存取集中在單一大型 Worker。部署單位可以整合多個模組，但原始碼責任仍須分離。

### 所有功能必須模組化

每個模組必須有單一責任、公開介面、明確依賴與擁有的資料範圍，不依賴其他模組的內部實作。

### 所有功能可獨立測試

模組必須能以受控輸入驗證輸出、錯誤、權限與邊界行為，不要求整個產品同時啟動才能測試。

### 所有 Engine 必須解耦

Engine 間以版本化 API、事件或 Queue 訊息協作。禁止跨 Engine 直接修改資料或共享可變內部狀態。

### 所有 API 可重用

API 契約不得綁定單一頁面或單一客戶。輸入輸出需一致、可版本化、可觀測，並具備明確的身份與租戶範圍。

### 禁止 Copy/Paste 開發

不得以複製整段既有功能建立新專案。若能力值得共用，應先定義模組契約，再由專案組合或擴充。

## Core 與 Project 邊界

- Core：共用契約、平台能力、通用政策與可重用模組。
- Project：客戶設定、品牌內容、專案流程、通路組合與部署設定。
- Adapter：外部供應商或通路的差異轉換，不把供應商格式擴散到 Engine。
- Extension：以公開介面擴充行為，不直接修改 Core 內部。

## API 標準

- 明確指定版本、認證、Tenant Context 與 Permission Scope。
- 使用一致的成功、錯誤、分頁與追蹤識別格式。
- 寫入操作需考慮冪等性、重試與稽核。
- 破壞性變更必須建立新版本與遷移計畫。
- 不得在 API 回應洩露其他租戶或內部實作細節。

## 品質與變更標準

- 先寫責任與驗收條件，再實作。
- 每次變更保持單一目的，避免順手重構不相關範圍。
- 新增依賴前先說明必要性、風險與替代方案。
- 必須具備單元測試、契約測試或符合風險的等效驗證。
- 日誌不得包含密碼、Token、完整個資或跨租戶資料。
- 文件、介面與實作必須同步版本化。
