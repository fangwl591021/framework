# Database Standard

## 文件範圍

本文件只定義資料概念與治理原則，不建立 Database Schema、欄位、索引或 Migration。

## 核心資料概念

### Identity

代表平台中的唯一自然人或系統主體，負責連結不同登入提供者；不等同於任何單一 Tenant 的 Member。

### Tenant

代表平台上的獨立組織與資料隔離邊界，擁有自己的成員、設定、權限與資源範圍。

### Member

代表 Identity 在特定 Tenant、Shop 或組織內的參與身份，可具有狀態、角色與領域屬性。

### Shop

代表 Tenant 內可獨立營運或提供服務的單位；其資料與權限仍受 Tenant 邊界管理。

### Transaction

代表可稽核的業務異動事實。交易資料需具備來源、時間、狀態、冪等識別與不可任意覆寫的歷史。

### Point

代表點數帳本與餘額衍生結果。每次增加、扣除、到期或調整都必須可追溯，不能只保存最終餘額。

### Event

代表平台內發生的領域事實，用於模組協作、通知、稽核或分析；事件需具備類型、版本、來源與時間。

### Relationship

代表 Identity、Member、Tenant、Shop 或其他資源之間具有方向、類型與有效期間的關係。

## 資料治理原則

- 每筆租戶資料都必須能判定 Tenant Scope。
- Identity 與 Member 分離，登入帳號不直接等於租戶會員。
- 每個 Engine 擁有自己的資料真相，其他 Engine 透過契約取得資訊。
- 交易、點數與權限異動必須具備稽核軌跡。
- 個資只在必要範圍保存，並定義保留、刪除與匿名化政策。
- 時間資料需保留明確時區語意，跨國顯示由應用層轉換。
- Schema 變更必須可版本化、可回復並先通過資料遷移評估。
- 分析用途不得繞過租戶隔離與個資政策。

## 未來 Schema 設計前置條件

1. 完成領域責任與資料擁有者定義。
2. 完成 Identity、Tenant 與 Permission 的存取情境。
3. 完成資料生命週期、稽核與法規需求。
4. 完成 API 與事件契約。
5. 完成容量、查詢與一致性需求評估。
